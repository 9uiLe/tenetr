# 0002: MVP の評価対象は num-path 日次完了画面、改善タスクは初期表示の削減と主操作の明確化

- Date: 2026-07-23
- Status: proposed(Human-only 決定。ユーザーの採用承認をもって accepted に更新する)
- Issue: #1

## Context

MVP は 1 画面・1 改善タスクに限定する(設計指示書 §4, §21 Phase 0)。Issue #1 は §23 の結果画面シナリオを既定候補と明記している。ただし対象アプリの実地調査(2026-07-23)で、§23.1 が仮定する要素(スコア・星・次へ・シェア)は現行の完了画面に存在しないことを確認した。実画面は「完了ヘッダー・連続記録カード・達成バッジ・翌日リマインダー・練習ボタン」で構成される。§23.1 の文言をそのまま使うと検証不能なタスクになるため、§23 の意図(すっきりさせ、次に進む操作を迷わせない)を実画面へ適用して具体化する。

対象アプリは private な個人開発物のため、本リポジトリには設計レベルの情報のみを記載する(AGENTS.md Security)。

## Decision

- 対象アプリ: num-path(iOS)
- 対象画面: 日次パズル完了画面(シナリオ ID: `completed`。アプリ既存のシナリオ注入機構の識別子に合わせる)
- 改善タスク(1文): 日次完了画面の初期表示を必要最小限にし、次の行動が迷わず1つに定まるようにする。完了の事実と連続記録は残す。
- 期待する After の方向性: 主操作が 1 つだけ視覚的に支配的で、補助情報(バッジ・リマインダー等)は視覚的重みが弱いか要求時開示になっている。人工的な誇張演出を加えない。

## Alternatives & dissent

- §23.1 の文言を無修正で採用する: 実画面に存在しない要素(スコア・星・シェア)を前提とし、検証チェックリストが恒久に満たせないため不採用。
- オンボーディング画面など別画面を対象にする: 状態再現に複数遷移が必要で Phase 2 の capture を複雑にする。完了画面には既存のシナリオ注入・スクリーンショット基盤があり再現コストが最小のため不採用。
- 実アプリに §23.1 の要素(スコア・星・シェア)を追加してから評価する: MVP の目的は Harness の成立であり、アプリ側の大規模改修は Out of scope のため不採用。

## Consequences

- Phase 0〜6 の検証はすべて `completed` シナリオを前提とする。
- Philosophy Pack 初期内容(Issue #2)の必須要素リスト・事例は実画面構成(完了ヘッダー・連続記録・主操作)に基づく。§23.2 の 5 原則は変更なしで適用可能。
- 設計指示書 §23.1 の例示と本決定の差分は本 ADR を正とする(指示書は改変しない)。
- 複数画面への展開は MVP 完了後に別 Decision で扱う。
